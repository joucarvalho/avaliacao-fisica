════════════════════════════════════════════════════════════
  PARTE 1AJ — Fix definitivo: div-swap de inputs durante captura
════════════════════════════════════════════════════════════

Sessão 2026-05-28. Branch ativa: ficha.

Abordagem anterior (lineHeight = height) confirmada ineficaz:
o html2canvas ignora line-height ao posicionar texto em <input>
— comportamento interno do motor, não configurável via CSS.

Solução: div-swap em capturarElemento() (pdf.ts).

Antes de chamar html2canvas, iteramos todos os <input type="text">
do elemento a capturar. Para cada um:
  1. Criamos um <div> com as mesmas classes CSS do input
  2. Copiamos os inline styles que já aplicamos (height, minHeight, etc.)
  3. Definimos display:flex + align-items:center no div
  4. Preenchemos div.textContent com input.value
  5. Substituímos o input pelo div no DOM (replaceChild)
  6. Armazenamos { div, input, parent } em inputRestores[]

No finally (antes de restaurar bgRestores):
  parent.replaceChild(input, div) → inputs voltam ao DOM
  O usuário nunca vê a troca — ocorre em milissegundos durante o export.

O html2canvas captura os divs (que renderizam texto corretamente via
flexbox) em vez dos inputs. O resultado é texto verticalmente centrado
em todos os campos — tanto nos h-6 do hero quanto nos h-9 das seções.

════════════════════════════════════════════════════════════
  PARTE 1AH — Corrige fontes ausentes no clone do html2canvas
════════════════════════════════════════════════════════════

Sessão 2026-05-28. Branch ativa: ficha.

Causa raiz identificada: textos desalinhados e espaçamentos
estranhos em todo o PDF exportado.

O html2canvas clona o documento antes de capturar. A função
patcharCloneOklch() percorria TODOS os <link rel="stylesheet">
e os removia, substituindo por <style> com CSS patchado.

O único stylesheet cross-origin é o Google Fonts (fonts.googleapis.com).
Por ser cross-origin, o CORS bloqueia a leitura das cssRules, então
ele ficava fora do Map cssOriginal. Resultado: o <link> do Google Fonts
era deletado do clone e substituído por um <style> VAZIO.

O html2canvas renderizava o clone sem DM Sans, IBM Plex Mono nem
Source Sans 3 — caía nos fontes do sistema operacional. Fontes
diferentes têm métricas diferentes (altura-x, ascendentes, entrelinhas),
o que causava:
  - textos deslocados para baixo nos inputs
  - espaçamento entre letras inconsistente
  - caracteres Unicode (₂) não suportados pela fonte do sistema

Duas correções em client/src/lib/pdf.ts:

  1. patcharCloneOklch():
     O loop que substitui <link> por <style> agora verifica se o href
     está em cssOriginal. Se não estiver (= cross-origin), faz `continue`
     e mantém o <link> original no clone. O Google Fonts link fica
     preservado — o html2canvas pode usar as fontes (já em cache do
     browser após o carregamento da página).

  2. capturarElemento():
     Adicionado `await document.fonts.ready` antes de chamar html2canvas.
     Garante que as três fontes estejam completamente carregadas e no
     cache do browser antes de qualquer captura. Na prática, resolve na
     primeira chamada e retorna instantaneamente nas demais (Promise settled).

════════════════════════════════════════════════════════════
  PARTE 1AF — Inline de CSS vars do hero para o html2canvas
════════════════════════════════════════════════════════════

Sessão 2026-05-28. Branch ativa: ficha.

Causa raiz identificada: Tailwind v4 usa var(--spacing) em TODAS as
utilities de espaçamento (py-3, px-4, h-6, mt-8, gap-3 etc.). O
html2canvas tem seu próprio parser CSS em JavaScript que NÃO resolve
var() — então padding, height e gap ficam em 0 internamente. O
background dos cards é desenhado com altura calculada errada, cortando
o conteúdo antes do padding inferior.

Fix em client/src/lib/pdf.ts, bloco do hero em exportarPDF:

  Antes de capturarElemento(hero), percorre hero + todos os filhos via
  [hero, ...hero.querySelectorAll('*')]. Para cada elemento:

  1. Itera LAYOUT_PROPS: height, minHeight, paddingTop/Bottom/Left/Right,
     marginTop/Bottom/Left/Right, rowGap, columnGap, borderRadius,
     letterSpacing, lineHeight.
  2. Lê o valor COMPUTADO via window.getComputedStyle(el)[prop] —
     o browser já resolveu var(--spacing) para valores em px.
  3. Se difere do inline style atual, salva o original em layoutRestores
     e escreve o valor computado como inline style.
  4. Também zera backdropFilter/webkitBackdropFilter via o mesmo array.

  No finally: restaura todos os inline styles de layoutRestores.

  O html2canvas lê inline styles com prioridade máxima (especificidade
  acima das classes CSS). Com os valores em px explicitamente definidos,
  o modelo interno do html2canvas tem as dimensões corretas.

════════════════════════════════════════════════════════════
  PARTE 1Z — Fix definitivo SecurityError Safari (html2canvas)
════════════════════════════════════════════════════════════

Sessão 2026-05-28. Branch ativa: ficha.

Causa raiz definitiva do SecurityError: The operation is insecure:

  A biblioteca html-to-image usa SVG <foreignObject> para renderizar HTML
  no canvas. O Safari (WebKit) marca o canvas como "origin-dirty" ao detectar
  qualquer canvas que recebeu um SVG com <foreignObject> — independente de
  crossOrigin, Blob URLs ou recursos externos. Isso está documentado na MDN:
  "Drawing an SVG image which includes a foreignObject element to a canvas
  makes the canvas origin-dirty." Chrome/Firefox são mais permissivos nesse
  ponto; Safari é estrito.

  As tentativas anteriores (Parte 1X e 1Y) eliminaram fetchAsDataURL e
  crossOrigin='anonymous', mas o erro persistia porque a taint vinha
  do próprio foreignObject — não de nenhum recurso externo. Confirmado
  pelo fato de o erro ocorrer mesmo com ficha completamente vazia (sem fotos).

Solução: substituir html-to-image por html2canvas.

  html2canvas renderiza cada elemento CSS manualmente (desenha retângulos,
  textos, bordas, sombras no canvas um a um), sem usar foreignObject. O
  Safari não marca o canvas como tainted nessa abordagem — canvas.toDataURL()
  funciona normalmente.

  Única limitação do html2canvas: não entende oklch() (cores do Tailwind v4).
  Solução: antes de capturar, lemos todos os estilos computados via
  window.getComputedStyle() — o próprio browser converte oklch para rgb()
  internamente — e aplicamos como estilos inline. Restauramos no finally.
  Resultado: html2canvas vê rgb() em todo lugar e renderiza corretamente.

Mudanças em client/src/lib/pdf.ts:

  - import { toSvg } de html-to-image → import html2canvas de html2canvas
  - Adicionada função normalizarCoresOklch(elemento):
      Walk todos os elementos filhos, lê getComputedStyle() para
      color, backgroundColor, borderColors, outlineColor, boxShadow etc.
      Aplica rgb resolvido como estilo inline. Retorna função de restore.
  - capturarElemento() simplificada:
      Chama normalizarCoresOklch + bgRestores antes da captura.
      Usa html2canvas(elemento, { scale:2, useCORS:false,
      allowTaint:false, foreignObjectRendering:false }).
      Retorna { dataUrl, largura, altura } — sem imagens overlay.
  - colocarGrupoNaPagina() simplificada:
      Sem loop de overlay de imagens (html2canvas renderiza fotos diretamente).
  - Removidas: aplicarPatchSafariSVG(), aguardarImagens(), ImageInfo interface.

Dependência nova:
  + html2canvas 1.4.1

pnpm check ✅ — zero erros de TypeScript.

  PARTE 1Y — Fix definitivo SecurityError Safari (crossOrigin + fontes)
════════════════════════════════════════════════════════════

Sessão 2026-05-28. Branch ativa: ficha.

Segunda rodada de correção do SecurityError: The operation is insecure
no Safari ao exportar PDF. A Parte 1X removeu os background-image
externos, mas o erro persistia.

Causa raiz identificada via leitura do código-fonte do html-to-image
v1.11.13 instalado (node_modules/html-to-image/es/util.js):

  export function createImage(url) {
    return new Promise((resolve, reject) => {
      const img = new Image();
      img.onload = () => { img.decode().then(() => {
        requestAnimationFrame(() => resolve(img)); }); };
      img.onerror = reject;
      img.crossOrigin = 'anonymous';  ← PROBLEMA
      img.decoding = 'async';
      img.src = url;
    });
  }

html-to-image seta img.crossOrigin = 'anonymous' ANTES de img.src.
No Safari, mesmo um Blob URL same-origin fica marcado como "origin-dirty"
quando crossOrigin='anonymous' está presente ao setar o src, fazendo

Correções aplicadas em client/src/lib/pdf.ts:

  1. aplicarPatchSafariSVG():
     Após criar o Blob URL (conversão do SVG data URL), chamamos
     this.removeAttribute('crossorigin') ANTES de setSrcOriginal.call(this, blobUrl).
     Isso limpa o atributo que html-to-image havia setado momentos antes.
     O Blob URL então carrega sem modo CORS → canvas NOT tainted → toDataURL() funciona.

  2. capturarElemento():
     Injetamos um <style> temporário antes do toJpeg:
       * { font-family: -apple-system, BlinkMacSystemFont, 'Helvetica Neue', Arial, sans-serif !important; }
     Isso força fontes de sistema durante a captura, eliminando qualquer
     referência a web fonts cross-origin (DM Sans, IBM Plex Mono etc.) no
     SVG gerado. Removemos o style no finally. O PDF ficará com fonts de
     sistema (ligeiramente diferente), mas o export funciona corretamente.

Defesa em profundidade — três camadas ativas:
  - Parte 1X: background-image externos → "none" antes da captura
  - Parte 1Y item 1: crossOrigin='anonymous' removido antes do Blob URL
  - Parte 1Y item 2: fontes de sistema forçadas no SVG

════════════════════════════════════════════════════════════
  PARTE 1H — CSP autoriza blob: e WASM (correção do pipeline)
════════════════════════════════════════════════════════════

Bug em produção pós-deploy da Parte 1G: o pipeline de redimensionamento
ficou bloqueado pela CSP em enforcement (Parte 1D). Console mostrava:

  Refused to load blob:... img-src directive of the CSP
  Refused to load blob:... worker-src directive of the CSP
  [ImageUpload] processamento de imagem falhou – Error: não foi possível
                                                          abrir a imagem

Causa: URL.createObjectURL() gera URLs blob:..., mas a CSP atual só
autorizava 'self', data:, CloudFront e R2 em img-src. Worker-src nem
existia explícito (herdava 'self' de defaultSrc). E heic2any (libheif-js)
é WASM, exigindo 'wasm-unsafe-eval' em script-src.

Mudanças mínimas necessárias na CSP (server/index.ts):

  - imgSrc            → adiciona "blob:"
  - workerSrc         → nova diretiva, ["'self'", "blob:"]
  - scriptSrc         → adiciona "'wasm-unsafe-eval'"

Análise de segurança:

  - blob: em img/worker é mesma origem (só JS da página gera blob URL).
    Padrão da indústria para apps que processam imagens client-side.
  - 'wasm-unsafe-eval' é o relaxamento mais sensível. Aceitável aqui porque
    o WASM vem do nosso próprio bundle (chunk lazy heic2any-XXX.js), não
    de origem externa. script-src continua sem 'unsafe-inline' e sem
    'unsafe-eval'. Pior caso prático: bug em libheif-js só causaria DoS
    local da aba, não vazamento.
  - default-src 'self', frame-ancestors 'none', object-src 'none' e
    base-uri 'self' permanecem intocados — CSP segue forte.

Arquivos alterados:

  - server/index.ts  → 3 ajustes na configuração da CSP do helmet, com
                        comentários inline explicando o motivo de cada um.
  - to-do.md         → tarefas U.1 a U.4 marcadas
  - git.txt          → este arquivo

━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  git status
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  O QUE FAZ:
    Mostra o estado atual dos seus arquivos — quais foram
    modificados, quais estão prontos para commit, quais
    o Git nem conhece ainda.

  QUANDO USAR:
    Sempre. É o primeiro comando que você roda antes de
    qualquer outra coisa. É inofensivo — só lê, não muda nada.

  SINTAXE:
    git status

  LEITURA DA SAÍDA:
    "Changes not staged"      → arquivo modificado, ainda não adicionado
    "Changes to be committed" → arquivo adicionado, pronto para commit
    "Untracked files"         → arquivo novo que o Git ainda não conhece
    "Deleted"                 → arquivo apagado (use git rm para confirmar)


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  git add
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  O QUE FAZ:
    "Seleciona" arquivos para entrar no próximo commit.
    É como colocar itens na sacola antes de fechar o pedido.

  SINTAXE — adicionar arquivo específico:
    git add nome-do-arquivo.tsx

  SINTAXE — adicionar múltiplos arquivos:
    git add arquivo1.tsx arquivo2.ts

  DICA:
    Prefira sempre adicionar arquivos pelo nome. É mais seguro
    e você aprende quais arquivos você tocou.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  git rm
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  O QUE FAZ:
    Avisa o Git de que um arquivo foi deletado e essa deleção
    deve entrar no próximo commit. Diferente do rm comum do
    sistema, o git rm registra a remoção no histórico.

  SINTAXE:
    git rm caminho/do/arquivo.ts

  QUANDO USAR:
    Quando você (ou o Claude) deletou um arquivo e quer que
    essa deleção fique gravada no commit. Equivalente a fazer
    git add num arquivo modificado, mas para arquivos apagados.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  git restore
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  O QUE FAZ:
    Joga fora as mudanças NÃO COMMITADAS de um arquivo e
    volta ele para o estado do último commit.

  ANALOGIA:
    É o "fechar sem salvar" do Git. Imagina que você
    rabiscou em cima de uma foto revelada — o Git tem o
    negativo guardado e te revela a foto de novo, limpinha.

  SINTAXE — restaurar um arquivo específico (recomendado):
    git restore caminho/do/arquivo.tsx

  SINTAXE — restaurar TUDO que está modificado:
    git restore .

    ⚠️  Cuidado com o "." — ele joga fora as mudanças de
    TODOS os arquivos modificados de uma vez. Prefira
    sempre restaurar arquivo por arquivo até ter certeza
    do que está fazendo.

  QUANDO USAR:
    - Bagunçou um arquivo e quer voltar pro último commit
    - Experimentou algo que não deu certo e quer desfazer
    - Salvou por engano e precisa "rebobinar"

  ⚠️  ATENÇÃO — É IRREVERSÍVEL:
    Mudanças não commitadas SOMEM. Não tem lixeira, não
    tem CMD+Z. Antes de rodar, SEMPRE faça:

      1. git status              → veja o que está modificado
      2. git diff <arquivo>      → veja o que vai ser perdido

    Se houver mudança que você quer manter, NÃO use restore
    — copie o conteúdo pra outro lugar antes.

  DIFERENÇA DE git checkout:
    Antes do git restore existir, usava-se "git checkout
    -- arquivo.tsx" pra mesma coisa. Ainda funciona, mas
    git restore é a forma moderna e mais segura (o
    checkout faz outras coisas além disso e era fácil
    confundir).


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  git commit
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  O QUE FAZ:
    Tira a "foto" dos arquivos que você selecionou com git add.

  SINTAXE:
    git commit -m "descrição do que foi feito"

  BOAS MENSAGENS DE COMMIT:
    ✓ "feat: adiciona upload de imagens via R2"
    ✓ "fix: corrige bug no auto-save prematuro"
    ✓ "feat(security): hardening completo"
    ✗ "mudanças"  ✗ "fix"  ✗ "atualizei coisas"


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  git push
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  O QUE FAZ:
    Envia os commits do seu computador para o GitHub.

  SINTAXE:
    git push origin ficha      → branch atual de desenvolvimento
    git push origin main       → branch de produção


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  git pull
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  O QUE FAZ:
    Baixa do GitHub mudanças que outros (ou outro computador
    seu) tenham feito. Sincroniza sua cópia local.

  SINTAXE:
    git pull origin ficha


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  RESUMO RÁPIDO — COLA
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  git status              → veja o que mudou
  git diff <arquivo>      → veja o que mudou linha a linha
  git add <arquivo>       → selecione o arquivo
  git rm <arquivo>        → registre uma deleção
  git restore <arquivo>   → descarte mudanças não commitadas
  git commit -m "msg"     → salve a foto
  git push origin ficha   → envie para o GitHub
  git pull origin ficha   → baixe do GitHub
  git log --oneline       → veja o histórico
  git checkout -b <nome>  → crie uma branch nova


════════════════════════════════════════════════════════════
  PARTE 3 — DEPLOY NO RAILWAY (variáveis de ambiente novas)
════════════════════════════════════════════════════════════

⚠️  CRÍTICO: NÃO faça push para main antes de configurar essas
    variáveis no Railway. Sem elas o servidor explode no boot
    com mensagem "JWT_SECRET não definido" / "AUTH_PASSWORD_HASH
    não definido" e a aplicação fica fora do ar até você
    configurar.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  VARIÁVEIS QUE PRECISAM ESTAR NO RAILWAY
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  JÁ CONFIGURADAS (não precisa mexer):
    DATABASE_URL          → string de conexão do PostgreSQL
    NODE_ENV              → "production"
    PORT                  → definido automaticamente pelo Railway
    R2_ENDPOINT           → endpoint do Cloudflare R2
    R2_ACCESS_KEY_ID      → credencial R2
    R2_SECRET_ACCESS_KEY  → credencial R2
    R2_BUCKET_NAME        → bucket R2
    R2_PUBLIC_URL         → URL pública das imagens

  NOVAS — A CONFIGURAR ANTES DO DEPLOY:
    AUTH_PASSWORD_HASH    → hash bcrypt da sua senha
    JWT_SECRET            → segredo aleatório de 256 bits que
                             assina os tokens JWT


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PASSO 1 — Gerar valores PARA PRODUÇÃO
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Recomendação de segurança importante:

  - AUTH_PASSWORD_HASH em prod PODE ser o mesmo do seu .env local
    (é a sua senha pessoal — não muda por ambiente).

  - JWT_SECRET em prod DEVE ser DIFERENTE do dev. Razão:
    se um dia o JWT_SECRET de dev vazar (ex: você compartilha o
    .env por engano), atacante poderia forjar tokens válidos em
    PRODUÇÃO. Mantendo separado, vazamento em dev fica isolado.

  Para gerar credenciais para produção:

    pnpm gerar-hash

  Quando ele pedir senha, digite a MESMA senha que você usa em dev
  (ou uma diferente, se quiser separar acessos). O hash vai sair
  diferente do dev (bcrypt usa salt aleatório), mas valida a mesma
  senha — pode usar qualquer um dos dois.

  COPIE o JWT_SECRET impresso. Esse SIM precisa ser diferente.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PASSO 2 — Adicionar as variáveis no Railway
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  PELA INTERFACE WEB (mais simples):

  1. Abra https://railway.app e entre na sua conta
  2. Selecione o projeto avaliacao_fisica
  3. Clique no serviço da aplicação (não o do PostgreSQL)
  4. Vá na aba "Variables"
  5. Clique em "+ New Variable"
  6. Adicione:
       Nome:  AUTH_PASSWORD_HASH
       Valor: $2a$12$....(o hash impresso pelo gerar-hash)....

       ⚠️  ARMADILHA COMUM: o output do gerar-hash imprime a linha
           inteira no formato "AUTH_PASSWORD_HASH=$2a$12$...", mas
           no Railway o campo "Value" recebe APENAS O VALOR — não
           inclua o nome nem o sinal de igual. Ou seja, cole somente
           o que vem DEPOIS do "=". O mesmo vale para JWT_SECRET.

  7. Clique em "Add"
  8. Repita para a JWT_SECRET com o valor impresso pelo script
  9. O Railway pergunta se quer "redeploy" — responda SIM apenas
     se já tiver feito git push origin main. Caso contrário,
     escolha "Save without deploy" e faça o push depois.


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PASSO 3 — Verificar o deploy
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Após o git push origin main e o Railway concluir o deploy:

  1. Abra a URL pública da aplicação no navegador
  2. Confirme que / redireciona automaticamente para /login
  3. Faça login com sua senha — deve abrir a tela de Alunos
  4. Tente uma senha errada — deve mostrar "senha incorreta"

  Se algo der errado (tela em branco, "Application failed to
  respond"), abra o painel do Railway → Deployments → último
  deploy → clique em "View Logs". Procure por:

    - "JWT_SECRET não definido"      → falta a variável JWT_SECRET
    - "AUTH_PASSWORD_HASH não..."    → falta a variável AUTH_PASSWORD_HASH
    - Outros erros                   → me avise na próxima sessão


━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  PASSO 4 — Como ROTACIONAR as credenciais (futuro)
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━

  Se em algum momento você suspeitar de vazamento (ex: alguém
  viu sua senha sobre o ombro, você commitou o .env por engano):

  TROCAR A SENHA:
    1. Rode pnpm gerar-hash localmente com a senha NOVA
    2. Atualize AUTH_PASSWORD_HASH no Railway com o hash novo
    3. Atualize AUTH_PASSWORD_HASH no .env local também
    4. Reinicie o servidor (Railway faz automaticamente)

  TROCAR O JWT_SECRET (invalida TODOS os tokens em uso):
    1. Rode pnpm gerar-hash apenas para gerar um JWT_SECRET novo
       (pode ignorar o hash da senha se não estiver trocando ela)
    2. Atualize JWT_SECRET no Railway
    3. Todos os usuários logados serão deslogados na próxima ação
       e precisarão fazer login de novo. Isso é proposital — é
       o efeito de "expulsar todo mundo" que você quer em caso
       de comprometimento.
